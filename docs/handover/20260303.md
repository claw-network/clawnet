### 1.2 registerDID ECDSA 签名修复

**问题**: `registerDID` 链上调用始终 revert，本地签名验证通过但链上失败。

**根因**: 已部署的 implementation 合约仍是旧版（4 参数，selector `0x0293ce52`），H-01 修复后的新版（5 参数含 `evmSig`，selector `0xeccf4570`）从未通过 UUPS 升级部署上去。

**修复**:
1. 创建了独立 UUPS 升级脚本 `packages/contracts/scripts/upgrade-identity.ts`（不依赖 OZ manifest）
2. 执行了两次 UUPS 升级：
   - 第一次：部署 H-01 implementation → `0x70cb634D...`
   - 第二次：追加 `adminSetController()` → 当前 impl `0xB3fFe0876349C60763F4dE6B4c337E2E0f2FCC85`
3. 签名消息格式：`abi.encodePacked("clawnet:register:v1:", didHash, controller)`

### 1.3 Per-DID EVM 地址隔离

**问题**: 所有 DID 共享同一个 node signer EVM 地址，余额无法区分。

**修复方案 — 确定性伪地址 + burn/mint 模式**:

1. **`deriveAddressForDid(did)`**（`identity-service.ts`）:
   ```
   "0x" + keccak256("clawnet:did-address:" + did)[last 20 bytes]
   ```
   为每个 DID 生成唯一的确定性 EVM 伪地址（无人持有私钥）。

2. **`ensureRegistered()`** 更新为使用派生地址作为 controller。

3. **`transfer()`**（`wallet-service.ts`）: 非 signer 地址使用 burn(from) + mint(to) 模式；signer 或 faucet 使用直接 `token.transfer()`。

4. **`transferRoutes`**（`transfers.ts`）: 发送方/接收方 DID 通过 `resolveDidToAddress()` 解析为 EVM 地址。

5. **链上**: 给 ClawIdentity 合约加了 `adminSetController(bytes32,address)` 函数（REGISTRAR_ROLE），用于迁移已有 DID。

6. **数据迁移**: 已有测试 DID `did:claw:zFy3Ed8bYu5SRHq5YK1YRz58iUpWxL27exCwngDwuH8gR` 的 controller 已更新为派生地址 `0x130Eb2b6C2CA8193c159c824fccE472BB48F0De3`，并 mint 了 1000 Token 到该地址。

### 4.1 需要清理

- [ ] **Geth debug API**: `/opt/clawnet/docker-compose.chain.yml` 中 `--http.api` 含 `debug`，调试完毕应移除并重建容器
- [ ] **服务器临时脚本**: `/tmp/test-register.cjs`, `/tmp/check-did-state.cjs`, `/tmp/migrate-did.cjs`, `/opt/clawnet/temp/debug-sig.cjs`, `/opt/clawnet/temp/test-ecrecover.cjs` — 可删除

### 4.2 已知不一致

- **Deployment 记录不一致**: `infra/testnet/prod/contracts.json` 使用的 proxy 地址与 `packages/contracts/deployments/clawnetTestnet.json` 不同（两次独立部署）。当前生产环境以 `infra/testnet/prod/contracts.json` 为准。
- **OZ Manifest**: `.openzeppelin/unknown-7625.json` 不包含 `0xee9B...` proxy 的记录。本次 UUPS 升级绕过了 OZ manifest，直接调用 `upgradeToAndCall`。后续升级需沿用同样方式或补齐 manifest。

### 4.3 架构注意事项

- **burn/mint 模式**: Per-DID 派生地址无人持有私钥，所以转账通过 node signer（持有 `MINTER_ROLE` + `BURNER_ROLE`）代为执行 burn(from) + mint(to)。这意味着 **node signer 是所有 DID 资产操作的受信任中介**。
- **DID 地址派生**: `deriveAddressForDid()` 是纯 keccak256 哈希裁剪，**一旦改变算法，所有已有余额将不可访问**。任何修改需配合迁移脚本。
- **faucet 特殊路径**: `transfer()` 中 `from === 'faucet'` 走直接 `token.transfer()` 路径（从 signer 余额扣）。
