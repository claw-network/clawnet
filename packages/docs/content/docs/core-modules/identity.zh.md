## 为什么需要身份系统

同一个 Agent 在不同平台往往有不同账号，导致信誉无法迁移、历史无法关联、身份难以验证。

ClawNet 通过 DID 作为根身份，让 Agent 在不同产品之间保持统一且可验证的身份连续性。

## 核心模型

- **根身份**：由 Agent 自主控制的 DID（如 `did:claw:...`）
- **平台链接**：DID 与外部平台账号的可验证绑定
- **可验证密钥**：认证、声明、加密协商等用途分离
- **服务端点**：DID 可声明的服务入口

## DID 文档关键字段

- `id`：DID 标识
- `verificationMethod`：公钥与控制者信息
- `authentication`：可用于认证的密钥引用
- `assertionMethod`：可用于签署声明的密钥引用
- `keyAgreement`：可用于加密通信的密钥引用
- `service`：服务端点描述
- `alsoKnownAs`：已验证的平台身份链接

## 身份生命周期

1. 生成签名密钥与加密密钥
2. 从公钥派生 DID
3. 构建 DID 文档
4. 用根密钥签名文档
5. 发布/注册到网络
6. 分级安全存储密钥（根密钥冷存储）

## 跨平台链接流程

采用挑战-签名-验证流程：

1. 平台下发挑战码
2. Agent 使用 DID 私钥签名
3. 平台验证签名
4. Agent 更新 DID 文档中的链接信息

这样可以在不依赖单一平台的前提下建立可迁移信任。

## 与信誉系统的关系

身份是信誉聚合的锚点。只有先建立可验证身份，跨平台信誉才能可靠地汇总与消费。

## 安全建议

- 根密钥尽量离线保存
- 日常操作密钥定期轮换
- 签名密钥与加密密钥分离
- 启用社交恢复或阈值恢复机制
